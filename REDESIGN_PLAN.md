# 召唤师档案重构实施规划

> 产品名：召唤师档案 · Summoner Atlas
>
> 定位：跨区域、跨模式的英雄联盟数据与版本决策平台
>
> 技术栈：Next.js 16、pnpm monorepo、TypeScript、shadcn/ui（Base UI）、Zustand、Hugeicons 、 tailwindcss
>
> 本文是后续 Codex 的实施依据。先完成迁移与验证，再替换线上入口。

---

## 1. 产品目标

召唤师档案不是只服务于大乱斗的工具，而是面向英雄联盟全体玩家的数据平台。它应支持不同服务器区域、不同游戏模式与不同研究深度：玩家可以快速查英雄强度，也可以研究版本、装备、符文、海克斯和阵容趋势。

### 1.1 一句话定义

> 看懂每个服务器、每个模式、每个版本的英雄联盟答案。

### 1.2 用户与任务

| 用户                 | 任务                                                 |
| -------------------- | ---------------------------------------------------- |
| 临场玩家             | 选人或对局开始前，快速查英雄、符文、装备与克制关系。 |
| 大乱斗与竞技场玩家   | 查英雄平衡调整、海克斯、装备与组合选择。             |
| 排位玩家             | 比较不同服务器和版本的英雄强度、出装与符文趋势。     |
| 内容创作者和研究用户 | 跟踪版本变化、地区差异与高分段玩法。                 |
| 管理员               | 维护版本、数据源、实体资料与用户权限。               |

### 1.3 支持范围

- 区域：国服、韩服、美服、台服、欧服，以及未来新增地区。
- 模式：召唤师峡谷、大乱斗、竞技场和未来模式。
- 对象：英雄、装备、符文、召唤师技能、海克斯、版本、位置、构筑和组合。
- 维度：版本、区域、段位、模式、时间范围、样本量与数据来源。

### 1.4 不可破坏原则

- 不删除任何已上线的核心查询、详情、账户或后台能力。
- 不因视觉重构改变指标定义或统计结果。
- 不把大乱斗专属概念当作全站默认结构。
- 不假设所有地区、模式和数据源具有相同字段或刷新频率。
- 不使用 Riot Games 官方商标、官方 Logo 或游戏素材作为品牌标识。

## 2. 现状审计与迁移基线

当前线上站点已具备英雄排行、海克斯排行、英雄详情、版本筛选、英雄搜索、主题切换、账户、绑定和后台数据管理等能力。重构必须逐项迁移并验证。

### 2.1 已确认能力

| 领域     | 当前能力                                           | 重构要求                                     |
| -------- | -------------------------------------------------- | -------------------------------------------- |
| 英雄排行 | Tier、胜率、登场率、场次、排序、定位筛选、版本选择 | 保留并增加区域、模式、可信度与版本变化。     |
| 英雄发现 | 中文名、拼音、外号搜索、英雄头像浏览               | 升级为全局搜索，保留全部搜索能力。           |
| 海克斯   | 海克斯排行、品质筛选、名称搜索、详情               | 作为模式能力呈现，不能污染峡谷模式信息架构。 |
| 英雄详情 | 平衡调整、技能、鞋子、加点、搭配、海克斯组合       | 改为结论优先，完整迁移数据。                 |
| 账户     | 登录、退出、个人中心、QQ/Riot 绑定                 | 保持权限和绑定行为等价。                     |
| 后台     | 版本、英雄、海克斯、装备、用户、角色管理           | 保留并增加数据源、区域与质量可观测性。       |

### 2.2 已发现问题

1. 首页同时展示英雄图库和完整长表，搜索、筛选与浏览的任务优先级不清。
2. 页面呈现为通用组件库表格，缺少游戏数据产品的品牌辨识和信息层级。
3. 详情页连续堆叠 Top 5、Top 15 表格，用户难以先获得可执行建议。
4. 紫、粉、青、黄、蓝色混用，没有稳定的数据语义。
5. 小字号、系统字体与非等宽数字降低表格扫描效率。
6. 移动端会出现版本标签竖排、筛选控件错位、桌面表格被硬压缩等问题。
7. 搜索与筛选不具备统一 URL 状态、清除入口和结果反馈。
8. 图像替代文本与部分交互标签缺乏语义，存在英文残留。
9. 初始 HTML 的 SEO 元信息不足，详情页分享能力需要重建。

### 2.3 迁移契约

在开发正式业务 UI 前完成：

- [ ] 列出全部路由、旧链接、URL 参数、权限角色和 API。
- [ ] 记录每个模式和区域可用的数据字段及数据来源。
- [ ] 记录每个页面的加载、空、错误、权限不足和未找到状态。
- [ ] 为搜索、筛选、排序、版本选择、绑定和后台写端到端验收用例。
- [ ] 建立新旧数据对账，相同条件下排名、胜率、登场率、场次必须一致。
- [ ] 为旧英雄与海克斯详情链接建立永久重定向。

## 3. 信息架构

### 3.1 核心上下文

所有查询由以下上下文驱动：

```text
区域 + 模式 + 版本 + 段位或分段 + 时间范围
```

默认值必须可解释。首次访问采用明确默认区域、当前稳定版本和推荐模式，并允许用户一键修改。区域选择不能隐藏在深层设置中。

### 3.2 站点地图

```text
/
├── /champions
│   └── /champions/[championId]
├── /items
│   └── /items/[itemId]
├── /runes
│   └── /runes/[runeId]
├── /augments
│   └── /augments/[augmentId]
├── /builds
├── /meta
├── /compare
├── /patches
│   └── /patches/[version]
├── /methodology

├── /login
├── /profile

│   └── /profile/preferences
└── /admin

    ├── /admin/regions
    ├── /admin/versions
    ├── /admin/champions
    ├── /admin/items
    ├── /admin/augments

```

### 3.3 导航

桌面端：

```text
召唤师档案 | 英雄 | 构筑 | Meta | 版本 | 全局搜索 | 区域/模式 | 账户
```

移动端：

```text
顶部：品牌 | 搜索 | 区域/模式 | 更多
底部：英雄 | 构筑 | Meta | 我的
```

联系、关注、捐赠、免责声明等放入更多菜单、支持页或页脚，不与核心查询争夺首屏位置。

### 3.4 首页

首页的工作是帮助用户选择任务，不是展示无限长的榜单。

1. 全局搜索，支持英雄、装备、符文、海克斯与别名。
2. 上下文选择器，显示区域、模式、版本和数据更新时间。
3. 当前版本摘要，展示上升最快、稳定强势、值得警惕的变化。
4. 英雄、构筑和模式专属数据的简短入口。
5. 区域差异与版本变化入口。
6. 数据方法、来源、免责声明和支持入口。

### 3.5 模式切换

区域和模式是一级上下文，而不是页面内小标签。

| 模式       | 重点模块                               |
| ---------- | -------------------------------------- |
| 召唤师峡谷 | 英雄、位置、符文、装备、对线、构筑。   |
| 大乱斗     | 平衡调整、海克斯、装备、技能、组合。   |
| 竞技场     | 双人组合、强化、装备、回合与搭配。     |
| 未支持模式 | 显示明确待接入状态，不能展示空白表格。 |

模式切换后，标题、筛选项、默认排序、指标解释和详情模块必须同步变化。

## 4. 页面规格

### 4.1 排行榜

统一结构：

```text
上下文栏：区域 / 模式 / 版本 / 分段 / 时间范围
筛选栏：搜索 / 位置或品质 / Tier / 最低样本 / 排序 / 清除
数据区：排名 / 实体 / 核心指标 / 版本变化 / 样本量 / 详情
```

要求：

- 所有筛选项同步至 URL，链接可直接分享和刷新恢复。
- 桌面端使用语义表格，表头吸顶，数字右对齐且使用等宽数字。
- 移动端改用数据行，名称、Tier、胜率和样本量始终可见。
- 大数据量使用虚拟列表。
- 低样本高胜率展示低可信度提示。
- 不以颜色作为 Tier、品质或涨跌的唯一表达。

### 4.2 英雄详情

页面从“数据堆叠”改为“建议、原因、证据”。

```text
英雄摘要
├── 区域、模式、版本、段位上下文
├── Tier、胜率、登场率、场次、版本变化
├── 当前模式平衡调整
└── 数据更新时间与可信等级

推荐方案
├── 召唤师峡谷：位置、符文、技能加点、核心装备
├── 大乱斗：平衡调整、技能、装备、海克斯
└── 竞技场：搭档、强化、装备、回合策略

深入数据
├── 推荐的完整证据
├── 全量构筑与组合
├── 版本趋势
└── 相关实体
```

要求：

- 模式不支持的模块不显示，不留空卡片。
- 每项推荐同时显示胜率、登场率、场次和可信度。
- 图像 `alt` 必须为真实英雄、技能、装备、符文或海克斯名称。
- 移动端通过锚点或标签分隔概览、构筑、组合和完整数据。

### 4.3 构筑与组合研究

流程：

1. 选择区域、模式和版本。
2. 选择英雄、位置或已有海克斯。
3. 选择一个或多个比较对象。
4. 展示推荐构筑、替代方案、样本量和差异。
5. 允许复制可分享链接。

核心流程必须在页面内完成，避免用多层对话框。

### 4.4 Meta、版本、账户与后台

- `Meta`：当前上下文的英雄强度、构筑趋势、区域差异和热门变化。
- `Patches`：版本说明、数据更新时间、版本前后涨跌和数据覆盖率。
- 个人中心：账号、已绑定账户、偏好和数据订阅。
- 后台：数据源、区域覆盖率、抓取状态、数据质量、版本、实体和用户权限。
- 所有写操作具备加载、成功、失败和可恢复反馈。
- 删除、解绑、角色变更使用 `AlertDialog`。

## 5. 视觉与交互系统

### 5.1 品牌方向

设计方向为“极地战术舱”：从符文、战术地图坐标、版本观测和峡谷地形获得灵感，但不复制 Riot 官方视觉资产。

现有 Logo 概念：

```text
public/brand/summoner-atlas-logo-concept.png
```

### 5.2 颜色 token

使用偏蓝的中性暗色作为默认产品场景，冰青色为主交互色，海克斯品质色仅服务于模式专属信息。

```css
:root {
  --background: oklch(0.17 0.015 250);
  --surface: oklch(0.22 0.018 250);
  --surface-raised: oklch(0.27 0.021 250);
  --foreground: oklch(0.94 0.008 240);
  --muted-foreground: oklch(0.7 0.018 245);
  --border: oklch(0.34 0.018 250);
  --primary: oklch(0.78 0.13 205);
  --positive: oklch(0.76 0.15 150);
  --negative: oklch(0.68 0.18 25);
  --warning: oklch(0.8 0.14 85);
}
```

禁止纯黑、纯白、彩虹数据图、渐变文字、通用紫蓝渐变与装饰性玻璃卡片。

### 5.3 字体、数据与动效

- 中文：`MiSans VF` 或 `HarmonyOS Sans SC`。
- 英文界面：`Satoshi` 或 `Geist`。
- 数值与版本：`Geist Mono`，使用 `font-variant-numeric: tabular-nums`。
- 标题使用 `clamp()`，正文行宽限制在 65 至 75 个字符。
- 交互反馈使用 160 至 220ms 的 `transform` 和 `opacity` 动画。
- 尊重 `prefers-reduced-motion`，核心数据页不加入重型滚动特效。

## 6. 技术架构

### 6.1 Monorepo

```text
apps/
├── web/                 # 公开站点和用户功能
└── admin/               # 只有独立部署需求明确时才拆出

packages/
├── ui/                  # Base UI shadcn 源码与业务组件
├── design-tokens/       # 主题和 token
├── domain/              # 区域、模式、版本、实体领域模型
├── data-access/         # API client、schema、query keys
├── analytics/           # 埋点
├── config/              # eslint、TypeScript、formatter
└── testing/             # fixtures、mock、共享测试工具
```

### 6.2 领域模型

`region`、`mode` 与 `patch` 必须是所有统计数据的一级字段。

```ts
type Region = "CN" | "KR" | "NA" | "TW" | "EUW" | "EUNE"
type GameMode = "summoners-rift" | "aram" | "arena"

interface DataContext {
  region: Region
  mode: GameMode
  patch: string
  tier?: string
  timeRange?: "24h" | "7d" | "30d"
}

interface RankedMetric {
  winRate: number
  pickRate: number
  matches: number
  confidence: "low" | "medium" | "high"
  previousPatchDelta?: number
}
```

不要在 UI 中以字符串临时拼接区域、模式、版本逻辑。用领域类型统一表达、校验和格式化。

### 6.3 Next.js、Base UI 与状态管理

- 使用 Next.js App Router。
- 数据查询优先 Server Components，搜索、筛选、表格交互使用 Client Components。
- 使用 `loading.tsx`、`error.tsx`、`not-found.tsx`。
- URL Search Params 保存上下文、筛选、排序和分页。
- 使用 Metadata API 提供动态 title、description、canonical、Open Graph 和 Twitter Card。
- 实体详情生成可分享 OG 图。
- 使用缓存 tag 精确失效区域、模式和版本维度的数据。

初始化 shadcn：

```bash
pnpm dlx shadcn@latest init -d --base base-ui
```

状态归属：

| 状态                                   | 位置                                    |
| -------------------------------------- | --------------------------------------- |
| 区域、模式、版本、筛选、排序、分页     | URL Search Params                       |
| 排行、详情、统计与缓存                 | Next.js 数据缓存，必要时 TanStack Query |
| 主题、密度、最近查看、组合研究临时选择 | Zustand                                 |
| 登录与表单校验                         | 服务端会话，React Hook Form + Zod       |

禁止把完整服务端排行榜复制到 Zustand。

### 6.4 业务组件

```text

RegionSelector
ModeSelector
VersionSelector
ChampionIdentity
ItemIdentity
RuneIdentity
AugmentIdentity
TierMark
MetricValue

```

Tier、可信度、数据格式化和实体展示逻辑由共享业务组件提供，禁止页面复制实现。

## 8. 上线门槛

### 功能与数据

- [ ] 当前所有核心功能均已迁移。

## 9. 工作规则

后续 Codex 必须遵守：

1. 先审计真实代码和 API，再修改业务逻辑。
2. 每一阶段只处理当前范围，不顺手重构无关模块。
3. 每次改动前确认是否影响迁移契约。
4. 先实现数据和响应式布局，再完成视觉细节。
5. 用真实数据验证，不使用假统计数字或 Lorem Ipsum。
6. 每个页面都考虑加载、空、错误、权限不足和未找到状态。
7. 每阶段至少运行 lint、类型检查、测试、构建和关键路由验证。
8. 每次 UI 改动至少检查 390px、768px、1440px 三个视口。

最低验证命令以实际 `package.json` 为准，至少覆盖：

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build
```
